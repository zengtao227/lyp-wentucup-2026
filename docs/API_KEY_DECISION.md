# API 密钥管理决策文档

## 🎯 核心问题

**是否需要将 API 密钥写入 YML 文件？**

---

## ✅ 最终决策：**不写入 YML**

### 理由分析

#### 1. **技术层面**

| 维度 | YML 文件的作用 | API 密钥的作用 |
|------|---------------|---------------|
| 用途 | Agent 配置（模型、工具、Prompt） | 外部 API 调用凭证 |
| 使用场景 | 导入到平台创建/更新 Agent | 通过 HTTP API 调用 Agent |
| 生效时机 | 导入时 | 运行时 |
| 关联方式 | 自动关联到账号 | 需要在请求头中携带 |

**结论**：YML 和 API 密钥是两个独立的概念，不应混在一起。

#### 2. **安全层面**

❌ **写入 YML 的风险**：
- YML 文件可能需要提交给组委会 → 密钥泄露
- YML 文件可能被其他参赛队伍看到 → 密钥被滥用
- YML 文件可能被提交到 Git → 永久记录在版本历史中

✅ **不写入 YML 的好处**：
- 密钥只存储在 `.env.local` 中 → 已加入 .gitignore
- 密钥只有团队成员知道 → 不会泄露
- 即使 YML 文件被分享，也不会暴露密钥

#### 3. **比赛场景分析**

| 场景 | 是否需要 API 密钥 | 说明 |
|------|------------------|------|
| 在 Playground 平台内测试 | ❌ 不需要 | 平台自动识别账号权限 |
| 评委在你的账号下测试 | ❌ 不需要 | 评委使用你的账号登录 |
| 提交 YML 给组委会 | ❌ 不需要 | 组委会导入后自动关联 |
| 外部小程序/网页调用 | ✅ 需要 | 需要在代码中配置密钥 |
| 答辩演示 Demo | ❌ 不需要 | 直接在平台上演示 |

**结论**：比赛的主要场景都不需要在 YML 中包含 API 密钥。

#### 4. **最佳实践参考**

参考业界标准（AWS、Azure、Google Cloud）：
- ✅ 配置文件（如 YML）只包含配置信息
- ✅ 密钥存储在环境变量或密钥管理服务中
- ✅ 代码和密钥分离，降低泄露风险

## 🎓 经验总结

### 核心原则

1. **配置与密钥分离**：配置文件（YML）和密钥（API Key）是两个独立的概念
2. **最小权限原则**：只在必要时使用 API 密钥
3. **安全第一**：宁可麻烦一点，也不要让密钥泄露

### 常见误区

❌ **误区 1**：认为 YML 文件需要包含所有信息才能运行
- **真相**：YML 只是配置，密钥是运行时凭证

❌ **误区 2**：认为 API 密钥是 Agent 的一部分
- **真相**：API 密钥是调用 Agent 的凭证，不是 Agent 本身

❌ **误区 3**：认为提交 YML 时必须包含密钥
- **真相**：平台导入 YML 后会自动关联账号，不需要密钥

### 最佳实践

✅ **实践 1**：使用环境变量存储密钥
✅ **实践 2**：将敏感文件加入 .gitignore
✅ **实践 3**：定期轮换密钥
✅ **实践 4**：监控 API 调用日志

## 📞 FAQ

### Q1: 如果评委需要测试，他们怎么获取 API 密钥？

**A**: 评委不需要 API 密钥。他们会：
1. 登录你的账号（或在你的账号下测试）
2. 直接在 Playground 平台上测试
3. 平台会自动识别权限，不需要额外配置

### Q2: 如果我要做一个外部小程序展示，怎么办？

**A**: 这种情况才需要 API 密钥：
1. 在小程序后端代码中配置 API 密钥（不要写在前端！）
2. 使用环境变量或配置文件管理密钥
3. 不要将密钥提交到 Git

### Q3: 如果密钥泄露了怎么办？

**A**: 立即采取以下措施：
1. 在 Playground 平台上删除旧密钥
2. 创建新的 API 密钥
3. 更新 `.env.local` 文件
4. 通知团队成员更新密钥

### Q4: 我可以创建多个 API 密钥吗？

**A**: 可以。建议：
- 开发环境一个密钥
- 演示环境一个密钥
- 生产环境一个密钥（如果有）

这样可以分别管理和监控，出问题时可以单独撤销。
