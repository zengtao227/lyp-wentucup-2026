这是什么

MCP = Model Context Protocol。简单说：

主办方已经帮你在酒店后台和比赛平台之间搭好了桥。 你的 Agent 不需要自己去对接酒店 PMS，只要让 Dify 调用这个 MCP 服务即可。

接入 MCP 后，路演时可以说：

「我们的 Agent 直接调用了比赛平台提供的酒店 MCP 服务，读取的是酒店后台实时房态，提交的订单会进入酒店预订系统。」

④ 景区门票价格的独立算术节点

这是什么

当前景区购票的价格计算是由 LLM 自己心算：prompt 里写了「成人票 88 元，儿童票 44 元」，让 AI 去算 3×88+1×44=?

问题：大模型算数偶尔会出错。你希望评委对价格数据有信心，就不应该让 AI 心算。

解决方案

在 Dify 工作流中加一个 Code 节点（代码执行节点），固定公式：

总价 = 成人票数 × 成人单价 + 儿童票数 × 儿童单价

代码节点是确定的——永远算不错。

是你自己加还是平台已经提供了？

是我加，你不用动手。 Dify 本身就提供 Code 节点功能，不需要主办方额外提供。我改 YAML 就行。

现在的流程 vs 加了算术节点后的流程

当前加了算术节点后
流程 票务查询 → LLM（在prompt里算价格）→ 确认摘要 票务查询 → LLM → Code 节点算总价 → 确认摘要
价格准确率 大模型心算，可能出错 代码计算，100% 准确

⑥ 参数提取器 + 变量赋值器

这是什么

当前预订提交 POST 的 payload 中有这样的字段：

"guest_name": "{{#conversation.guest_name#}}"

如果 conversation 变量里没有 guest_name，发出去的就是空字符串，甚至更糟——把 {{#conversation.guest_name#}} 这串文字直接发出去。

参数提取器解决什么

它解决「怎么把 LLM 说的文本变成程序能用的变量」。具体：

  1. LLM 输出确认摘要后，参数提取器从摘要文本中提取「姓名=张三」「入住日期=5月15日」「房型=豪华套房」
  2. 写入 Dify 的对话变量
  3. POST 节点读变量时就有真实数据了

变量赋值器又是什么

比参数提取器更简单的 Dify 原生功能。你直接告诉它：「这个变量 = 那个节点的输出」。不需要写提取规则。

是否需要你做

不用你动手，我来改 YAML。 但导入后可能需要你确认一下参数提取规则配置是否正确。

⑦ Dify 控制台自动导入脚本

现状

我改完 YAML → 提交到 GitHub → 你手动打开 Dify 网页 → 导入 YAML → 发布。全部手动操作。

脚本做什么

写一个脚本自动完成上述手动步骤：

  1. 从浏览器中安全获取登录 session
  2. 调用 /console/api/apps/imports 自动导入 YAML
  3. 调用 /console/api/apps/imports/{id}/confirm 确认
  4. 调用 /console/api/apps/{id}/workflows/publish 发布

为什么是 P3

因为涉及登录凭证（cookie/session），需要额外注意安全性。且比赛期间版本迭代不频繁，手动导入完全够用。
