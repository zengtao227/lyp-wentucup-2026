# 小程序支付与统一价格源设计记录

日期：2026-05-09

## 当前结论

比赛系统中，客房支付与景区活动/门票支付都应通过酒店自有小程序完成。Bot 负责咨询、推荐、收集信息和生成待支付订单入口，不负责直接完成支付，也不应在支付前宣称“已支付成功”。

两个小程序入口：

- 客房预订/支付：`#小程序://小途乐订/2SQD7GEkqfdCiOw`
- 景区活动/门票：`#小程序://小途乐游/UalQXnmvHDjPJ1s`

两个小程序都是酒店自己的小程序，并且都包含比赛演示用的 `20000` 元虚拟余额。

## 支付边界

### 客房支付

客房支付只走「小途乐订」小程序。

推荐流程：

1. 用户通过 Bot 咨询和预订客房。
2. Bot 收集入住日期、退房日期、人数、房型、联系人、联系电话。
3. Bot 生成预订申请或待支付订单编号。
4. Bot 引导用户打开「小途乐订」小程序。
5. 用户在小程序内使用虚拟余额完成支付。
6. 支付成功状态以小程序为准。

Bot 可以说：

```text
您的客房预订申请已生成，请打开「小途乐订」小程序，凭订单/流水编号完成支付。比赛演示账户内含 20000 元虚拟余额。
```

Bot 不应说：

```text
您已完成支付。
确认短信已发送至您的手机。
电子预订凭证已生成。
```

除非后续拿到小程序订单查询或支付回调 API，并能确认支付状态。

同理，在当前 workflow 只调用 jsonplaceholder mock POST 并返回 `id` 的情况下，Bot 只能说“预订申请已提交、酒店将回电核实、请到小途乐订完成支付”。它不能宣称短信已发送、电子凭证已生成、订单已最终生效或房间已锁定。

### 景区活动/门票支付

景区活动和门票只走「小途乐游」小程序。

推荐流程：

1. 用户通过 Bot 咨询周边景区、天气、适合活动。
2. Bot 帮用户选择景区/活动、游玩日期、人数。
3. Bot 生成或引导到待支付订单。
4. Bot 引导用户打开「小途乐游」小程序。
5. 用户在小程序内使用虚拟余额完成支付。
6. 支付成功后，由「小途乐游」小程序生成门票或核销凭证。

重要边界：

- Bot 不负责生成门票。
- Bot 不应说“门票已生成”。
- Bot 只应说“请到小途乐游小程序完成支付，支付成功后由小程序生成门票/凭证”。

## 统一价格源设计

用户提供的后台截图显示，`问途数字化营销管理平台 v2.0` 中存在「客房预订 - 房价管理」页面，可以按年份、月份、日期维护房价。

目标设计：

```text
数字化营销管理平台
  ├─ 房型设置
  ├─ 配额管理
  ├─ 价格体系 / 房价管理
  ├─ 客房订单
  └─ 旅游商城订单
        ↓
统一价格 API / 数据库
        ↓
网页版价格展示
小途乐订客房价格
小途乐游景区活动价格
Bot 价格查询
```

原则：

1. 价格不应写死在 Bot/YAML 里。
2. 员工只在数字化营销管理平台的价格管理页面维护价格。
3. 网页版、小程序和 Bot 都应读取同一个后台价格源。
4. 若某一天或某个时间段价格变化，员工只改一次后台价格，所有入口同步更新。
5. Bot 在无法读取价格源时，不应编造价格，只能引导用户到小程序或前台确认。

### 携程价格参考口径

携程酒店页面可作为人工设置价格时的市场参考：

- `https://hotels.ctrip.com/hotels/437229.html?cityid=316`

但携程页面价格受日期、库存、活动、会员等级和订单结算页影响，不适合作为 Bot 直接硬编码的实时价格源。页面上的价格说明也强调：展示价格可能随实时更新和市场波动变化，最终成交价格以订单结算页为准。

因此当前建议：

1. 后台人工定价时可以参考携程同日期、同房型价格区间。
2. Bot 不直接抓取或承诺携程价格。
3. Bot 在没有统一价格 API 前，应说“价格以小途乐订/小途乐游小程序实时显示为准”。
4. 后续若拿到数字化营销管理平台的价格查询 API，应优先查询酒店自己的后台价格源，而不是抓取携程页面。

## 当前待验证问题

目前还不能确认「网页版」「小途乐订」「小途乐游」和 Bot 是否已经读取同一套后台 API/数据库。需要通过实际测试确认。

### 建议测试：客房价格同步

1. 在数字化营销管理平台的「房价管理」页面，选择一个测试日期和房型。
2. 把价格设置成一个明显数字，例如 `666`。
3. 打开网页版同日期、同房型，查看价格是否变成 `666`。
4. 打开「小途乐订」小程序同日期、同房型，查看价格是否变成 `666`。
5. 如果网页版和小程序同时同步，说明它们大概率读取同一套后台价格源。
6. 再尝试抓取或确认该价格查询 API，后续让 Bot 也读取同一价格源。

### 建议测试：景区价格同步

1. 在数字化营销管理平台的「旅游商城」或相关活动/门票价格页面，选择一个景区活动和测试日期。
2. 把价格设置成一个明显数字，例如 `88` 或 `123`。
3. 打开网页版旅游商城同活动，查看价格是否同步。
4. 打开「小途乐游」小程序同活动，查看价格是否同步。
5. 如果同步，再确认 Bot 能否通过同一接口读取景区活动价格。

## 对后续 YAML 的影响

后续 v1.7/v1.8 应按以下方向修正或增强：

1. 客房支付入口必须使用「小途乐订」。
2. 景区活动/门票入口必须使用「小途乐游」。
3. 景区门票确认文案不能引导到「小途乐订」。
4. Bot 不能声称已付款，除非有支付状态查询 API。
5. Bot 不能声称已生成景区门票，门票应由「小途乐游」支付成功后生成。
6. 若能拿到统一价格查询 API，Bot 应查询后台实时价格。
7. 若暂时拿不到价格 API，Bot 应说“价格以小程序实时显示为准”，并打开对应小程序。

## 可公开性判断

可以写进公开 YAML/GitHub：

- `#小程序://小途乐订/2SQD7GEkqfdCiOw`
- `#小程序://小途乐游/UalQXnmvHDjPJ1s`
- 支付流程说明
- 虚拟余额为比赛演示账户能力

不建议写进公开 YAML/GitHub：

- 高德真实 API Key
- Dify Service API Key
- 控制台登录 token/cookie
- 未公开的后台管理账号或接口鉴权凭证

高德 key 即使是比赛一次性 key，也不能放入公开数据仓库。当前主仓库已改为 private，v1.9 起为了测试便利嵌入了比赛一次性 key；赛后应删除或轮换。
