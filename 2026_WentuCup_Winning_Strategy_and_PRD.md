## 🤖 PART 3：核心技术栈最佳实践（2026 版）

### 3.1 模型路由策略：双脑驱动架构

| 场景 | 推荐模型 | 理由 |
| :--- | :--- | :--- |
| 意图分类、快速路由 | Qwen-Turbo | 响应速度快，成本低，分类稳定 |
| 标准问答、礼貌回复 | Qwen-Plus / ERNIE-3.5 | 中文语言质量高，合规性好 |
| 复杂场景（议价、投诉处理、行程规划） | DeepSeek-V3 / DeepSeek-R1 | 逻辑推理能力强，适合多步骤任务 |
| 真正的推理链任务（需展示思维链） | DeepSeek-R1 | 可输出 reasoning_content，适合答辩演示 |

**注意**：`DeepSeek-R1` 才有 `reasoning_content` 字段，`DeepSeek-V3` 没有。当前 lyp.yml 中会议宴会节点用了 V3 却取 `reasoning_content`，这是 Bug（见 PART 1）。

### 3.2 知识库构建最佳实践

**Chunk 策略**：
- 酒店基础信息、政策类文档：按段落切分（512 tokens），保留结构
- 周边景点、餐厅推荐：按条目切分，每条目保持独立语义
- 高频问题库：Q&A 对格式，直接匹配原文

**检索策略**：
- 启用 Reranking（已在 lyp.yml 中配置，注意部分节点未启用）
- top_k 设为 4-6，对于安全性要求高的问题（退款政策、价格）降低 top_k 至 2-3 并提高精确度

**幻觉防控**：在所有涉及价格、可用性的 Prompt 中加入：
```
重要约束：只能基于以下上下文信息回答，若上下文中未提及，必须回复"请联系前台确认"，严禁自行编造数字或承诺。
上下文：{{#context#}}
```

### 3.3 提示词工程最佳实践

**当前问题**：lyp.yml 中所有节点使用相同的 system prompt，缺乏差异化角色定义。

**改进方案**：为每个分支设计专属角色 Prompt：

```
# 会议宴会服务专属 Prompt
你是【酒店宴会销售顾问】，具备以下专业能力：
- 熟悉各宴会厅容量、布置方案（鱼骨/剧院/U型）、餐标范围
- 能够根据人数和预算主动推荐最优方案组合
- 洞察企业客户的隐性需求（如：是否需要投影/签到台/特定餐饮定制）
- 对于超出标准的需求，给出有吸引力的"可协商"信号而非直接拒绝

行动原则：每次回复都要包含至少一个"价值确认"或"下一步建议"
```

### 3.4 对话记忆设计

在 Dify 中启用 `conversation_variables` 存储关键用户状态：

```yaml
conversation_variables:
  - name: guest_name          # 宾客姓名（首次获取后无需重复询问）
  - name: check_in_date       # 入住日期
  - name: party_size          # 人数
  - name: special_needs       # 特殊需求（过敏/无障碍/安静楼层）
  - name: budget_range        # 预算范围（用于定向推荐）
  - name: sentiment_flag      # 情绪状态（normal/frustrated/delighted）
```

## 🎖️ PART 4：OPC 视角下的多智能体协同架构

本项目构建基于 **"一人民宿主 AI 助理团"** 的四位一体智能体集群，回答核心命题：

> *一位民宿主，如何用这套 AI 系统，一个人服务 50 位客人，同时运营营销、预订、接待、售后？*

### 4.1 智能体角色重定义

| 角色名称（OPC视角） | 核心职责 | Dify 实现方式 |
| :--- | :--- | :--- |
| **智能前台（Smart Receptionist）** | 全渠道咨询入口，意图识别，情绪感知，自动分流 | 问题分类器 + 情绪判断分支 |
| **销售顾问（Sales Assistant）** | 主动推荐增值服务（早餐、接机、门票），识别升级销售时机 | 独立分支 + Upsell 触发 LLM 节点 |
| **行程助手（Trip Planner）** | 整合周边资源（高德/大众点评），生成个性化行程推荐 | HTTP工具节点 + 外部 API |
| **售后管家（After-sales Butler）** | 处理投诉、退改签请求，生成标准化工单 | 投诉分支 + 知识库（退改政策）|

### 4.2 核心演示场景：OPC 民宿主的一天

**场景设计（适合现场 Demo）**：

> 一位民宿主，手机上收到三个并发咨询：
> 1. 客人A：问能否提前入住（需查房态）
> 2. 客人B：询问周边推荐，顺带问有没有接机服务
> 3. 客人C：对房间异味表达不满

AI 同时处理三条对话：
- 客人A → 智能前台 → 知识库查政策 → 自动回复"可提前至13:00，需加收X元"
- 客人B → 行程助手 → 生成周边推荐 → 销售顾问介入推荐接机套餐（**主动 Upsell**）
- 客人C → 情绪感知识别投诉 → 售后管家给出致歉+解决方案 → 输出工单

**这个场景完整展示了 OPC 的核心价值**：一人公司 × AI 团队 = 零人力成本并发服务。

### 4.3 Upsell 智能触发逻辑

基于国际最佳实践（Revinate 2026），以下触发词应激活 Upsell Agent：

| 触发信号 | 推荐动作 | 转化率参考 |
| :--- | :--- | :--- |
| "我们是来过周年纪念日的" | 主动推荐浪漫套餐/鲜花布置 | 较高（设计目标，需测试集验证） |
| "带孩子" | 推荐亲子加床、儿童早餐、门票套餐 | 较高（设计目标） |
| "出差" | 推荐商务早餐、停车位、快速退房服务 | 中（设计目标） |
| "我对酒店的第一印象" | 情绪正面 → 追加推荐周边高评分餐厅 | 中（设计目标） |
| 询问退房时间 | 推荐 Late Check-out 服务 | 中低（设计目标） |

> **答辩注意**：上述"转化率"为设计目标，答辩时说明"我们设计了测试集来验证，实测数据以测试结果为准"。Revinate 2026 报告称 AI 个性化推荐整体可提升非房费收入 15%-25%，可作为行业背景引用（但注意：该数据指整体收益提升，非单条触发词转化率）。

## 🏆 PART 5：需求规范（PRD）

### 5.1 功能性需求

#### F1：多意图并行识别（核心亮点）
- 用户在单次输入中包含多个需求时，系统能识别并拆解为多个子任务
- 示例：`"明天早上8点叫醒，帮我订一辆车去机场，另外昨天的房间太吵了"`
- 系统拆解：①叫醒服务请求 ②用车服务推荐 ③投诉记录与处理

#### F2：情绪感知与差异化响应
- 在问题分类前增加情绪检测节点（Prompt 实现）
- `正常态`：标准礼貌回复
- `不满/投诉态`：优先道歉、提供补偿方案、减少推销
- `愉悦/感谢态`：适度进行增值服务推荐

#### F3：主动 Upsell 引擎
- 在所有完成标准回答的节点后，增加"推荐钩子"
- 规则：当对话轮数 ≥ 2 且情绪为正面时，在回复末尾添加一句个性化推荐
- 示例：`"顺便提一句，您入住期间我们餐厅明天有独家海鲜主题晚宴，提前预约享85折～"`

#### F4：适老化语音支持
- 开启 `speech_to_text: enabled: true`
- 在 Opening Statement 中提示："支持语音输入，点击麦克风图标即可说话"
- 大字号友好型回复格式：避免长段落，使用分点/编号

#### F5：个性化记忆
- 利用 `conversation_variables` 记录关键偏好（见 3.4 节）
- 在同一对话中，第二次问到相关信息时主动引用：`"您之前提到带着两个孩子，我们亲子房还有2间..."`

### 5.2 非功能性指标

| 指标 | 目标要求 | 依据 |
| :--- | :--- | :--- |
| 意图识别准确率 | ≥ 90% | 通过测试集评估，答辩时展示 |
| 首轮回复响应时间 | ≤ 3 秒 | Dify 流式输出，用户体验基准 |
| 任务自动完成率 | ≥ 85% | 释放人力 40%，Canary AI 2026 基准 |
| 非房费收入贡献 | +15% ~ 25% | Upsell Agent 转化，Revinate 2026 数据 |
| 多轮对话上下文保持 | 10 轮窗口 | Dify memory window 配置 |
| 幻觉拒答率（价格/政策） | 100% | 所有价格类问题必须走知识库 |

## 💡 PART 6：2026 国际酒店 AI 最佳实践参考

> 以下实践来自 Conduit AI、Revinate、HospitalityNet、Lighthouse 2026 年研究报告，可在答辩中作为行业背景引用。

### 6.1 Voice AI：从通信工具到服务大脑

**国际趋势**：2026 年，Voice AI 成为"直订防线"——每一通未接来电都是一次流失的直接预订。

**实践要点**：
- AI 语音客服转化率优于传统语音客服（来源：Conduit AI 2026，具体数字答辩时请以官方报告原文为准）
- 关键能力：实时多语言（英/中/日/韩）、语速自适应、情绪识别
- 在中国市场：优先接入微信语音消息通道（用户习惯）

**在竞赛中的落地**：
- 开启 Dify STT，在 Demo 时展示语音输入 → 文字处理 → 语音回答的完整链路
- 答辩 PPT 中引用数据：声网 RTC + LLM 构建的 Voice Agent，已被首旅如家等头部连锁采用

### 6.2 Hyper-Personalization：从群体到个体

**研究数据**：实现个性化卓越的企业，收入比同行高 40%（Lighthouse 2026）。

**两层个性化**：
- **会话内个性化**（短期）：用 conversation_variables 捕获当次对话中的偏好信号
- **历史个性化**（长期）：通过知识库存储 VIP 客户档案，AI 在回复时自动检索

**触发词个性化示例**：
```
当用户提及"周年纪念"→ 推荐浪漫套餐
当用户提及"商务出行"→ 推荐商务增值服务
当用户提及"带宠物"→ 主动说明宠物政策并推荐宠物友好楼层
```

### 6.3 多渠道统一（Omnichannel）

**国际最佳实践**：统一平台同时处理微信/网站/电话/OTA 咨询，确保体验一致性。

**Dify 实现路径**：
1. 通过 Dify API 接入微信公众号
2. 同一工作流服务不同渠道，通过 `sys.channel` 变量差异化回复格式（微信 vs 网页）

### 6.4 AI 收益管理：向"虚拟 GM"进化

**核心算法**（用于答辩展示）：

$$\Delta P_{\text{建议}} = f(\underbrace{D_{\text{未来需求预测}}}_{\text{节假日/赛事日历}}, \underbrace{C_{\text{竞对定价}}}_{\text{OTA实时爬取}}, \underbrace{I_{\text{剩余库存}}}_{\text{PMS接口}})$$

**实际可演示的简化版本**：
- 在知识库中录入"节假日定价规则"（元旦/春节/黄金周）
- AI 根据用户询问的日期，自动给出"当前最优价格档位"建议
- 在答辩中展示：AI 识别到"国庆节前三天"时，自动推送"早订优惠"话术

### 6.5 Agent-to-Agent 趋势（2026 前沿）

**行业方向（答辩加分项）**：未来宾客侧 AI 助理（如用户手机上的 AI 秘书）将与酒店侧 AI 系统直接进行机器对话，完成预订谈判。这要求酒店系统提供**机器可读的标准化接口**（结构化 JSON 而非自然语言）。

**在答辩中的讲法**：
> "我们今天的 Chatbot，只是第一步。当宾客侧 AI 普及后，我们系统将进化为接受 AI-to-AI 通信的服务端，实现零人工干预的全自动预订闭环。"
